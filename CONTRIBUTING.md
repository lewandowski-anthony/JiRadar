# Contributing to Jiradar

Thank you for your interest in contributing to Jiradar. This document provides a step-by-step guide on how to fork the repository, set up your local development environment, and submit a pull request.

---

## 1. Fork the Repository

To establish an isolated environment for your modifications without impacting the main project, you must create a fork:

1. Navigate to the main Jiradar repository page on your hosting platform.
2. Click the "Fork" button located in the interface header.
3. Choose your personal account or organization workspace as the destination namespace. This instantiates a copy under your direct control.

---

## 2. Clone and Configure the Local Workspace

Clone your newly created fork to your local machine and map the original repository as the upstream source provider for future synchronization:

```bash
# Clone the fork to your local directory
git clone [https://github.com/lewandowski-anthony/jiradar.git](https://github.com/lewandowski-anthony/jiradar.git)
cd jiradar

# Configure the tracking reference to the original source repository
git remote add upstream [https://github.com/ORIGINAL_OWNER/jiradar.git](https://github.com/ORIGINAL_OWNER/jiradar.git)

# Verify remote tracking assignments
git remote -v
```

---

## 3. Local Initialization Sequence

Before making any changes, compile the project dependencies to activate the local repository compliance validation scripts managed by Husky:

```bash
# Enter the frontend environment directory
cd front-end

# Pull down dependencies and register the local quality gate pre-push hooks
npm install
```

---

## 4. Create a Feature Branch

Always isolate your work within structured development branches derived from the updated upstream main tracking state:

```bash
# Force local main branch alignment with remote upstream updates
git checkout main
git pull upstream main

# Initialize a dedicated, descriptively named development branch
git checkout -b feature/your-feature-name
```

---

## 5. Local Verification and Quality Gates

Implement your changes within the backend or frontend directories. Prior to committing code, execute local test suites to guarantee compliance with the workspace requirements.

### Backend Validation (Spring Boot)
Ensure that Clean Architecture code bindings align, integration suites compile, and JaCoCo engine reports demonstrate system stability:
```bash
cd ../back-end
mvn clean verify
```

### Frontend Validation (Angular & Vitest)
Verify that all TypeScript, template HTML structures, and test suites conform to style guidelines and test coverage metrics:
```bash
cd ../front-end
npm run lint
npm run test:coverage
```
*Note: Local structural code changes must maintain an absolute 80% coverage minimum metric target across statement, branch, line, and functional execution graphs to pass the verification gates.*

---

## 6. Commit and Push Changes

Stage your verified modifications, construct descriptive change records using semantic commit guidelines, and publish the feature stream back to your personal remote fork:

```bash
# Track and lock updated structural components
git add .

# Log a clear semantic commit declaration
git commit -m "feat: user metric calculation optimization for parallel issues tracking"

# Transmit localized change states to your personal tracking branch
git push origin feature/your-feature-name
```
*Note: Executing the push command automatically triggers the localized `.husky/pre-push` compilation lifecycle hook script. The workflow process initiates Maven structural code execution passes alongside full frontend Vitest coverage extraction before granting branch authorization.*

---

## 7. Open a Pull Request

Finalize the contribution workflow by generating an explicit platform review request tracking item:

1. Open your fork dashboard page on the web interface.
2. Select the "Compare and pull request" option generated by recent branch data transmissions.
3. Assert that the comparison target boundaries evaluate correctly from `lewandowski-anthony/jiradar (feature/your-feature-name)` mapping into the foundational line of `ORIGINAL_OWNER/jiradar (main)`.
4. Document a clear summary profile outlining the problem scope, resolved programmatic areas, test performance coverage validations, and component dependency impacts.
5. Select "Create pull request" to route the proposal into the integration evaluation lifecycle queue.
